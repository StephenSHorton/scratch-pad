I've been thinking a lot about how to test the extraction pipeline faster. The problem with live meetings is they take real time. Even a five-minute test meeting blocks five minutes of my afternoon, and I usually need to run multiple iterations to tune a prompt change. So the realtime pacing constraint is the main thing slowing down iteration on the graph mutation loop.

Transcript import sidesteps that entirely. If we drop the realtime pacing for offline sources, we can chew through a thirty-minute fixture as fast as the model returns. That's maybe two or three minutes of LLM calls instead of thirty minutes of conversation. The unit tests are still the right tool for narrow correctness checks, but for whole-pipeline behavior — does the extraction actually produce a coherent graph from a real-shaped transcript — there's no substitute for running the actual pipeline end to end.

The other thing I want to keep clean is that the import path goes through the same extraction code as live. No separate offline branch. If a regression in mutateGraph breaks live, it should break import too, which means import is also a regression test. Adding a parallel "offline-only" path defeats that.

One open question is whether the synthesized timestamps are good enough. The current rule is four hundred milliseconds per word, minimum one second per chunk. That gives about a hundred and fifty words per minute, which is roughly normal speaking rate. I think that's fine for now; if it turns out to bias the batch boundaries in some weird way, we can tune it. But I'd rather ship and find out than over-engineer the timestamp synthesis up front.
